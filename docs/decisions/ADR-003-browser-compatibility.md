# ADR-003：浏览器数据加载采用 XMLHttpRequest 兼容路径

- 状态：Accepted
- 日期：2026-07-12

## 背景

Production 页面在开发电脑正常，但在另一台电脑、iPhone 和 iPad 上出现静态框架可见、数据为空和交互卡死。服务端 API、Supabase 和公网 JSON 响应均正常。

## 问题

旧前端使用现代 JavaScript、`fetch`、异步 Promise 链和 `Response.text()`。异常设备能够收到 200 响应头，但执行链停在读取响应正文，未进入 JSON 解析和渲染。连续请求、单一聚合请求、Buffer、Content-Length 和关闭连接都未单独解决。

隔离页证明异常设备上的原生 XMLHttpRequest、JSON.parse 和简单 DOM 渲染均正常；随后 ES5 分阶段页面的 A–F 全部通过。

## 方案

考虑过继续调整 fetch 时序、继续修改 HTTP 传输、引入前端框架，以及建立最小 ES5/XHR 路径。由于证据已把故障定位到旧主页面执行链，选择最后一项。

## 最终决定

- 正式浏览器入口使用 ES5 语法和 XMLHttpRequest 请求 `/api/dashboard-compat`。
- 服务端兼容接口完整序列化并一次性返回聚合数据。
- 默认隐藏诊断面板；`?debug=1` 启用 XHR 和 A–F 分阶段诊断。
- 不再把旧 fetch/Response.text 路径作为首页关键加载路径。

## 影响

### 正面

- 已在原异常设备完成数据加载与渲染验收。
- 诊断能力仍可按需开启，不污染正式页面。
- 单次聚合请求减少首页加载链路。

### 代价与约束

- 代码风格需保持 ES5 兼容，开发体验不如现代语法。
- 最终证据说明是前端执行链兼容问题，但没有证明某一个浏览器内核缺陷；后续不得把推断写成确定事实。
- 引入现代构建工具前必须在既有异常设备上回归测试。
