# Future：Admin 管理后台

状态：Design only，未开发、未进入当前 API 或数据库。

## 目标

为授权的数据管理员提供独立后台，安全维护指标、分类、宏观事件、来源和历史记录，减少直接数据库操作。

## 目标用户

- 数据管理员
- 市场数据维护人员
- 具备审计权限的运维或负责人

普通只读用户不应自动获得管理能力。

## 功能范围

### 登录

- 受控身份认证、Session 过期和退出。
- 支持角色与最小权限。
- 登录失败、异常设备和高风险操作可审计。
- 不在浏览器代码中保存数据库或 Service Role Secret。

### 数据录入

- 新增指标、宏观事件和未来数据源配置。
- 字段校验、来源必填、单位与变化口径提示。
- 缺失数据使用“待录入”，禁止伪造数值。
- Production 写操作需要更高确认级别。

### 编辑

- 编辑当前值、前值、日期、来源、频率和展示属性。
- 保存前展示字段差异。
- 支持乐观锁或版本检查，避免覆盖他人修改。

### 删除

- 默认使用停用/归档，不直接物理删除有历史或引用的数据。
- 高风险删除要求二次确认、权限和审计理由。
- 批量删除必须有预览、备份和回滚能力。

### 分类

- 管理分类名称、顺序、首页代表性指标和显示状态。
- 分类变更不应破坏历史数据或 API 兼容性。

### 历史记录

- 查看每次数据修改的操作者、时间、旧值、新值和原因。
- 支持按 symbol、用户和时间筛选。
- 审计记录不可由普通管理员修改。

## 权限模型预留

| 角色 | 建议能力 |
| --- | --- |
| Viewer | 只读 |
| Editor | 新增和编辑授权数据 |
| Approver | 审批高风险修改或发布 |
| Admin | 用户、角色、分类和系统配置 |
| Auditor | 只读审计记录 |

最终权限模型需要单独 ADR 和安全评审。

## 数据模型预留

可能需要：`users`、`roles`、`user_roles`、`audit_logs`、`indicator_history`、`categories`。当前 schema 中不存在这些表；具体设计必须遵循 [数据库扩展原则](../11-DATABASE.md#扩展原则)。

## API 预留

未来管理 API 应与公开读取接口分离，并具备认证、授权、CSRF/重放保护、速率限制、输入验证和审计。不得直接暴露数据库凭据。

## 环境与发布

- 先在 Staging 建立测试账号和隔离数据。
- Production 默认关闭未完成的 Admin 入口。
- 所有权限和删除功能必须进行安全测试。
- 发布遵循 [测试规范](../13-TESTING.md) 与 [发布规范](../06-RELEASE.md)。

## 非目标

- 本文不实现登录、页面、API 或数据库。
- 不决定具体身份供应商。
- 不授权当前公开页面执行高风险管理操作。
