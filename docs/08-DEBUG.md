# Debug 手册

本文规定故障定位方法，并记录 2026 年浏览器兼容问题的经验。架构背景见 [ADR-003](decisions/ADR-003-browser-compatibility.md)，部署与日志入口见 [运维手册](09-OPERATIONS.md)。

## 排查原则

### 不要猜测

先收集可复现证据，再提出假设。HTTP 200 只证明收到响应头，不证明正文读取、JSON 解析或 DOM 渲染成功；页面框架可见也不证明主脚本执行完成。

### 一次只验证一个假设

每轮只改变一个变量，例如请求方式、响应正文、渲染函数或浏览器。一次修改多个变量会让成功和失败都无法解释。

### 使用二分法定位

把执行链拆成边界：网络、接口、正文、解析、标准化、渲染、监听器。先判断故障位于哪一半，再继续细分，直到找到第一处未完成步骤。

### 先证明问题，再修改代码

优先使用只读检查、最小测试页和可见日志证明停止位置。没有证据时不进行架构重写或连续“试试看”式修复。

### 修改后立即验证

每次修改都要在原始异常设备上验证。开发机器通过不能替代第二台电脑、手机和平板结果。

### 保持可回滚

调试提交范围要小，记录假设和结果。临时诊断工具必须有启用条件、清理条件和安全边界。

## 推荐排查流程

```text
确认环境
   ↓
确认网络
   ↓
确认接口
   ↓
确认浏览器能力
   ↓
确认可见日志
   ↓
二分缩小执行范围
   ↓
实施最小修复
   ↓
原设备回归测试
   ↓
Staging 验收
   ↓
Production 冒烟验证
```

## 分阶段方法

### 1. 确认环境

- 记录 URL、`APP_ENV`、分支、commit 和部署时间。
- 检查问题只发生在 Production、Staging 还是本地。
- 确认设备、系统、浏览器名称与版本、代理/VPN 状态。
- 使用 `/api/health` 确认环境和数据库连接，不记录连接串。

### 2. 确认网络

- 页面和静态 JS/CSS 是否返回 200。
- 请求是否真正发出、Pending、取消或被缓存。
- 对比异常设备与正常设备的状态、响应大小和耗时。
- 不因某台机器正常就排除网络或代理差异。

### 3. 确认接口

- 在同一异常设备直接访问或请求 API。
- 检查状态码、Content-Type、正文长度和 JSON 结构。
- 分开证明“响应头到达”和“正文读取完成”。
- 对写接口只在 Staging 使用测试数据。

### 4. 确认浏览器能力

- 建立不加载业务脚本和框架的最小测试页。
- 分别验证脚本启动、XHR/fetch、正文、JSON.parse 和 innerHTML/appendChild。
- 使用 ES5 最小实现区分语法兼容与业务初始化问题。

### 5. 确认可见日志

- 日志必须显示在异常设备能看到的位置，不能只依赖开发机 Console。
- 每一步记录 start、success、error 和最后成功位置。
- 捕获同步错误、异步拒绝、超时和中止。
- 错误至少包含 name、message、stack；不得空 catch。

### 6. 二分缩小范围

示例分段：

```text
A：只更新数据基准
B：只渲染硬编码卡片
C：只渲染接口第一条数据
D：渲染全部市场总览
E：增加宏观事件
F：恢复导航、刷新和维护监听器
```

找到首个失败阶段后，只检查该阶段及其输入，不重复排查已证明正常的数据库或 API。

### 7. 修复与回归

- 修复应针对已证实边界，不扩大架构范围。
- 在 Staging 检查正常设备和原异常设备。
- 验证数据、交互、滚动、按钮、空状态和错误状态。
- 正式收尾时默认关闭调试，但保留安全的按需诊断入口。

## Debug Checklist

### 环境和日志

- [ ] 是否确认当前是 Production、Staging 还是本地
- [ ] 是否记录部署 commit
- [ ] 是否使用 `?debug=1` 开启页面诊断
- [ ] 是否查看 Render Log
- [ ] 是否查看 Browser Console 和 Network
- [ ] 是否记录浏览器版本、设备和代理状态

### 服务和数据

- [ ] 是否验证 `/api/health`
- [ ] 是否直接验证相关 API
- [ ] 是否区分响应头成功与正文读取成功
- [ ] 是否确认 JSON 数量和关键字段
- [ ] 是否确认当前连接的是正确环境数据库

### 浏览器覆盖

- [ ] 开发电脑 Chrome
- [ ] 第二台电脑 Chrome
- [ ] Edge
- [ ] iPhone Safari
- [ ] iPhone Chrome
- [ ] iPad Safari
- [ ] 有代理与无代理场景（如与问题相关）

### 发布回归

- [ ] Staging 已复现或验证修复
- [ ] Production 未被测试数据污染
- [ ] 修复后页面数据、导航、刷新和维护功能正常
- [ ] Debug 默认关闭且按需可开启
- [ ] Render 服务端无新增错误

## 浏览器兼容事件复盘

### 已观察事实

1. 开发机器正常，其他电脑、iPhone 和 iPad 数据为空。
2. `/api/indicators`、`/api/events` 和数据库均正常。
3. 旧主页面执行链停在 fetch/Response 正文读取附近。
4. 调整请求时序、聚合响应、Buffer 和 Content-Length 没有单独解决。
5. 最小兼容测试页上的 XHR、正文、JSON.parse 和 DOM 渲染全部成功。
6. ES5/XHR 分阶段 A–F 在异常设备全部成功。

### Lessons Learned

- **不要一次修改多个变量**：否则无法判断哪个变化有效。
- **Debug 页面价值很高**：最小页面能快速区分网络、浏览器基础能力和业务代码。
- **保留兼容测试页面**：至少在 Staging 和多设备验收期间保留；正式环境长期暴露前需确认不泄露数据和内部信息。若最终下线页面，应保留本手册、测试步骤和可快速恢复的模板。
- **先定位再修复**：证明第一处失败边界后再改代码。
- **可见日志优于只写 Console**：移动设备未必方便打开开发者工具。
- **正常设备不能代表全部用户**：兼容修复必须回到原异常设备验证。
- **不要把推断写成事实**：本事件证明了旧前端执行链兼容问题，但未证明某个单一浏览器内核缺陷。
